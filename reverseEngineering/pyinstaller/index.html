# 前言

最近接触到一个程序，大概功能就是用激活码安装游戏，有点好奇想看看什么原理

# 一、查壳脱壳

使用 Detect It Easy 进行查壳

原来是用 PyInstaller 打包的，这不简单？都不用反汇编了，直接反编译

# 二、反编译

先使用 PyInstaller Extractor 工具解包

新版 PyInstaller Extractor 会自动对 pyc 文件添加 magic head，所以这里我们不用手动修改，也可以检查一下以防万一
然后安装 uncompyle6 反编译 pyc 文件

uncompyle6 不支持 Python310，那试试 pycdc 反编译工具

JUMP_IF_NOT_EXC_MATCH 是最近刚添加的字节码， pycdc 尚未进行适配，因此生成的源码是不完整的，可以用 pycdas 反汇编，结合 main.py 来进行分析
但是呢， JUMP_IF_NOT_EXC_MATCH 字节码与 try...except 异常处理相关，我的目标是功能分析，所以要看的是正常无报错的代码，所以我们可以忽略这个错误
不过因为 pycdc 报错后会中断跳过当前函数，所以要对 pycdc 进行修改
打开 pycdc Github 仓库利用 Github 的搜索功能，搜索 Unsupported opcode，搜索到关键代码在 ASTree.cpp 当中

虽然说我不会 C++ ，但是基本的是可以看得懂，因为用的方法都一样，这里是用了 swtich...case...default 方法，当 switch 的值与所有的 case 的值不相等时，就会运行 default 语句，所以这里我们要修改 default 之后的代码，fprintf 除外全部注释掉，然后让 opcode 出栈，从上下文分析可以发现 opcode 是存在 stack 里的，在 fprintf 的下一行写入 stack.pop ();
重新编译得到 pycdc.exe，重命名为 pycdc1.exe 和原版 pycdc 区分开，方便以后调用
使用 pycdc1 对 main.pyc 反编译，并找到关键函数

找到 on_click_act 函数，从函数名和代码可以明显看出，这是按下按钮后执行的函数，首先判断 code 是否为空，是则弹出窗口提示 "请输入激活码"，所以 code 就是激活码，接下来就是更新按钮和输入框的状态，然后调用 WorkThread 函数
从 PYZ-00.pyz_extracted 文件夹找到 work_thread.pyc，用 pycdc1 反编译，仍然有 Unsupported opcode: JUMP_IF_NOT_EXC_MATCH 报错，不管它，直接看代码

在主函数里，调用 main_process 函数并传递了 code，在 main_process 函数声明 result 变量的值为 xxxcore.query_game (code)，看来 xxxcore 就是我们要找的
反编译，报错，管它呢...

因为报错，后面的代码变得诡异，没关系，这不是关键代码
可以看到这个函数就是从 mysql 取出与游戏有关的信息，解读 sql 命令，查找 dlcid，appid 等数据，并且要满足 game_info.appid 和 activation_code.appid 相等，以及 activation_code.code 等于输入的 code 的条件
这里就省略看代码的过程，直接概括

  • 在 /config/config.vdf 的 depot 元素添加用于解密 manifest 的 key
  • 获取作者在腾讯云存储的 manifest 文件，添加到 /depotcache/ 下
  • 获取 User32.dll (同上)，存放在程序根目录
  • 修改 /xxx.cfg 添加 BootStrapperInhibitAll=enable，查了一下，这个参数用来阻止自动更新
  • 如果存在 /hid.dll，就删除
  • 删除 /appcache/ 下的所有文件，清除 app 缓存，可能是为了刷新仓库界面
  • 在 / Applist / 下 (如果没有就创建) 添加 txt 文件，文件名是有序的数字，如 "0.txt","1.txt","3.txt"，文件内容为 appid、dlcid、manifest，一个文件只写一个 id
    由于本期的主题是逆向 pyinstaller 打包的程序，所以关于 User32.dll 的逆向放到下期
